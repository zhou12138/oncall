06 · 典型场景一

无 API ERP / 内网系统：不改系统，也能让 AI 进入流程

以前：人工操作链路

  • 员工登录内网 ERP
  • 手动查询、导出 Excel
  • 复制整理、汇总报表
  • 发送给管理层

现在：Agent 调度 Worker

  • Agent 调用 MCPHub
  • Gateway 调度 ERP Worker
  • Worker 在内网机器执行查询/导出
  • 返回结构化结果和文件
价值：不改 ERP、不暴露内网、不搬账号密码，先跑通低风险业务闭环。

08 · 差异化

不是 MCP Server、不是远程桌面、不是传统 RPA

方案解决什么局限LandGod / MCPHub 区别
企业 MCP Server把已有 API 暴露给 Agent前提是已有 API调度机器、权限、环境和本地工具
远程桌面人控制一台电脑给人用，不是给 Agent 用给 Agent 的工具调用和治理层
传统 RPA固定脚本自动化流程变动后脆弱Agent 动态决策 + 工具调度
每机 Agent每台机器一个大脑成本高、治理分散一个大脑调度多个手脚
MCP Server 连接 API 化世界；LandGod / MCPHub 连接还没 API 化的真实企业世界。

✅

工具白名单

Agent 只能调用授权工具，不直接远控整机。

👤

人工审批

高风险操作进入 approval gate。

🧾

审计回放

记录调用方、Worker、参数、结果与 artifact。

🚨

熔断控制

可禁用 token / worker / tool，快速止损。

✓Agent 不直接进入内网，不直接拿 ERP、UKey、Office、DB 权限。
✓Worker 在本地原环境中按最小权限执行，敏感登录态不离开机器。
✓Gateway 统一鉴权、路由、审批、审计、追踪和熔断。

11 · 结论与请求

先让 AI 在一个真实流程中跑起来，再扩展成企业级执行网络

核心结论

LandGod / MCPHub 不是远程控制工具，而是企业 AI Agent 的执行基础设施。它让 AI 在不重构旧系统、不搬走敏感权限的前提下，进入真实业务流程。

需要领导支持

  • 选择 1-2 个低风险高价值 PoC 场景
  • 确定试点机器和业务负责人
  • 明确安全审批边界
  • 给 2-4 周验证周期
不用等 API 改造，让 AI 从“辅助回答”进入“真实执行”。
